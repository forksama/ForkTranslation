# 调研档案 004：学马仕论坛串翻译输入输出格式与知识库组织方案

- 档案编号：Research-004
- 建档日期：2026-07-13
- 状态：规格草案
- 主题：标准输入文件 A、翻译输出 B、PR 字幕输出 C、领域知识库目录与强 agent 翻译流程

## 1. 本档案定位

Research-001 已经说明了论坛串翻译为什么需要作品上下文、社区上下文、稳定知识和事件知识。本档案在此基础上更新为更接近实际生产的方案：用户先通过油猴脚本手工选择需要的楼层，并为楼层打上“楼主、回复者1、回复者2”等角色标签；随后强 agent 读取标准化文本文件 A 和领域知识库，输出两类译文：

1. **翻译版 B**：基于 A 的完整中文译文，尽量保留楼层结构、角色标签和讨论顺序，适合人工审校。
2. **PR 字幕版 C**：基于 B 再做语义拆分和字幕切分，便于后续脚本导入 Premiere Pro。楼主正文中的叙述句作为 `旁白`，角色台词作为具体角色；非楼主回帖者也作为独立角色。

本仓库应被视为多领域翻译知识库，而不是只服务一次论坛串的临时目录。`gakumasu` 应拆成独立领域目录；未来如果加入其他作品、社区或题材，也应按领域并列扩展。

## 2. 翻译产品优先级

本项目的“忠实”按以下优先级理解：

1. **忠实于人物口吻**：优先保留角色、发帖者或叙述者的语气、情绪强度、说话节奏、距离感和故意不自然之处。
2. **忠实于中文互联网阅读体验**：在不改变含义的前提下，让中文读者读起来像自然的中文社区内容或自然的视频字幕。
3. **忠实于字面**：保留原文信息、指代和逻辑，但不机械照搬日语语序。

同时有三条硬约束：

- 不“加戏”：不新增原文没有的事实、心理活动、笑点、铺垫或解释。
- 不擅自删减：被选入 A 的楼层原则上都应在 B 中有对应译文，在 C 中也应有对应字幕段；确实不适合上屏的内容必须标明原因。
- 不替原帖作者纠错：如果原帖误读剧情或表达含混，译文应保留这种状态，必要时在译注或审校备注中说明。

## 3. 标准输入文件 A

### 3.1 A 的目标

A 是油猴脚本从网页中抽取并经人工选择、打标后的纯文本文件。它不需要保存完整 HTML，但必须让强 agent 稳定知道以下信息：

- 这是什么领域、站点和论坛串；
- 选中了哪些楼层；
- 每个楼层的楼号、角色标签、原文正文和引用关系；
- 哪些内容是正文，哪些内容是正文内引用楼号、人工备注或缺失信息；
- 同一角色标签在整个文件中保持一致。

A 不应提前翻译，也不应为了中文读者润色日文原文。清洗只限于去掉网页按钮、广告、导航、无关 UI 文本等噪音。

当前あにまん油猴脚本只做纯文本 ETL：不导出图片、缩略图、图片链接、OCR 或媒体说明。若某次翻译的笑点依赖图片，应由人工在 A 中另行补充 `editor_note`，或在后续版本中扩展媒体字段。

### 3.2 A 的推荐文件格式

A 建议使用 UTF-8 Markdown 文本，并包含 YAML front matter 与楼层块。文件名建议为：

```text
source-A.md
```

推荐结构如下：

````markdown
---
schema: fork-thread-source/v1
domain: gakumasu
source_site: animanch
source_url: "https://bbs.animanch.com/board/0000000/"
thread_title: "原串标题"
capture_date: "2026-07-13"
source_language: ja
selected_by: userscript
role_policy: "楼主固定为 OP；非楼主按人工标注为 回复者1、回复者2..."
---

# thread

## post: P0001
floor: 1
role: 楼主
kind: post
reply_to: []
reply_to_floors: []
time:
tags: [opening]
source_url:

```ja
ここに原文をそのまま入れる。
改行はできるだけ保存する。
```

## post: P0002
floor: 5
role: 回复者1
kind: reply
reply_to: [P0001]
reply_to_floors: [1]
time:
tags: []
source_url:

```ja
返信本文をここに入れる。
```

## post: P0003
floor: 8
role: 回复者2
kind: reply
reply_to: []
reply_to_floors: []
time:
tags: []
source_url:

```ja
別の返信本文をここに入れる。
```
````

### 3.3 A 的必填字段

- `schema`：固定为 `fork-thread-source/v1`，便于后续脚本识别版本。
- `domain`：本项目中使用 `gakumasu`。
- `source_site`：例如 `animanch`。
- `source_url`：原串地址；如果是缓存或镜像，应另行标注。
- `thread_title`：原串标题。
- `capture_date`：抽取日期，使用 `YYYY-MM-DD`。
- `post`：每个楼层的稳定 ID，建议 `P0001`、`P0002` 递增，不必等同于楼号。
- `floor`：原始楼号。
- `role`：人工角色标签，如 `楼主`、`回复者1`、`回复者2`。
- `kind`：楼层类型，如 `post`、`reply`、`narration`、`deleted-note`。
- `reply_to`：引用或回复目标中已经被选入 A 的楼层，填对应 `post` ID；如果引用目标未被选中则留空。
- `reply_to_floors`：正文中出现的原始引用楼号，如 `>>22`。这是从网页稳定可取的引用信息，即使引用目标未被选中也应保留。
- fenced code block：楼层原文正文，语言标记使用 `ja`。

### 3.4 A 的可选字段

- `time`：楼层发布时间；没有就留空，不要伪造。
- `tags`：如 `opening`、`spoiler`、`meme`、`has-reference`、`uncertain-speaker`。
- `source_url`：单楼链接；如果脚本能生成则保留。
- `quote`：被引用内容。当前油猴脚本默认不生成该字段；如人工补充，最好包含被引用楼号、角色和短引文。
- `media`：图片、动图、视频或 OCR 文本。当前油猴脚本默认不生成该字段；如笑点依赖图片，应由人工另行补充。
- `editor_note`：人工抽取者备注，只写事实性说明，例如“此处原图失效”“该楼疑似接上一串”。

### 3.5 A 的角色标注规则

`role` 在 A 中只表示“论坛楼层作者标签”，不是字幕里的语义说话者。建议遵守：

- `楼主` 只用于原串楼主或本次翻译中被指定为主发帖者的人。
- 非楼主统一使用 `回复者1`、`回复者2`、`回复者3` 等，不要混用“路人A”“观众B”等别名。
- 如果无法确认两个楼层是否来自同一人，不要强行合并成同一个回复者。
- 如果同一楼层内部出现虚构角色台词，A 仍先把该楼层归属于发帖角色；这些内部角色由 B 到 C 的阶段基于语义理解再拆分。
- `旁白` 不应作为 A 的普通楼层角色。它是 C 中用于叙述句、场景句和必要标题转场的字幕语义角色。

## 4. 翻译输出 B

B 是“根据 A 仅进行翻译后的内容”。它用于人工审校和追溯，不承担字幕切分任务。文件名建议为：

```text
translation-B.md
```

B 应保持 A 的楼层顺序和 `post` ID，每个楼层至少包含：

```markdown
## post: P0002
floor: 5
role: 回复者1
reply_to: [P0001]

原文：
> 返信本文をここに入れる。

译文：
回复正文放在这里。

备注：
- 如无必要可省略。
```

B 的规则：

- 不为字幕长度强行断句；优先保证译文准确、自然、可审校。
- 保留楼层、角色、引用关系和疑点。
- 对楼主正文中的 `角色名「台词」`、叙述句、作者说明应尽量保留清晰边界，便于 C 阶段拆成具体角色、旁白或楼主说明。
- 对原文错字、梗、反讽、语气异常应尽量在译文中体现；无法兼顾时在备注标注。
- 不把注释混进译文正文。译注、存疑和考据应放在 `备注` 或独立审校区。
- 如果 A 的某楼被判断为不应翻译，必须写明 `译文：〔不译，原因：...〕`，不能静默删除。

## 5. PR 字幕输出 C

### 5.1 C 的目标

C 是给 PR 导入脚本读取的机器友好格式。它不是完整文章，而是一组按播放顺序排列的字幕 cue。每个 cue 表示一次屏幕展示，默认只放 1 行；只有当这句话的含义尚未表达完毕且单行实在过长时，才拆成 2 行。

文件名建议为：

```text
pr-subtitles-C.jsonl
```

推荐使用 JSONL，而不是 Markdown 表格。原因是字幕文本可能包含标点、引号、换行和竖线，JSONL 更容易被脚本稳定解析。

### 5.2 C 的基本结构

每行一个 JSON 对象。第一行是元数据，其后每行是一个 cue。cue 的颗粒度不是“每个短句一条”，而是“一次屏幕展示一条”：同一语义角色的一小段完整意思可以放在同一个 cue 中。cue 不能过长，也不要过短；默认应是一个可独立阅读的小意思单元。`lines` 默认只放一个字符串；只有单行过长且意思不能自然拆成下一条 cue 时，才使用两个字符串表示两行。

```jsonl
{"type":"meta","schema":"fork-pr-subtitle/v1","domain":"gakumasu","source":"translation-B.md","title":"原串标题"}
{"type":"cue","id":"C0001","role":"旁白","source":["thread.title"],"lines":["原串标题的中文译名"],"notes":""}
{"type":"cue","id":"C0002","role":"旁白","source":["P0001"],"lines":["去年解散的中等部顶级组合 SyngUp!，她就是队长贺阳燐羽。"],"notes":""}
{"type":"cue","id":"C0003","role":"学园长","source":["P0001"],"lines":["没错，就是那位贺阳燐羽。"],"notes":""}
{"type":"cue","id":"C0004","role":"学P","source":["P0001"],"lines":["原来如此，大概能想象。"],"notes":""}
{"type":"cue","id":"C0005","role":"回复者1","source":["P0009"],"lines":["本来有点警惕，结果是正常燐羽。"],"notes":""}
{"type":"cue","id":"C0006","role":"学园长","source":["P0001"],"lines":["她和黑井的契约好像结束了，","可引退演唱会还没办成。"],"notes":"two_line:long_sentence"}
```

字段说明：

- `type`：`meta` 或 `cue`。
- `schema`：固定为 `fork-pr-subtitle/v1`。
- `id`：字幕段 ID，建议 `C0001` 递增。
- `role`：用于 PR 脚本匹配角色样式。它是字幕语义角色，不等同于 A 的论坛楼层作者标签。
- `source`：来源 post ID 或特殊来源，如 `thread.title`。
- `lines`：字幕行数组，默认 1 个字符串；最多 2 个字符串。每个字符串是一行字幕。
- `notes`：脚本可忽略的备注；用于标记拆分、疑点或人工处理需求。

### 5.3 C 的角色规则

- `旁白`：主要用于楼主正文中的叙述句、场景句和心理描写，也可用于串标题或必要转场。
- 作品角色名：楼主正文中出现 `角色名「台词」` 时，C 的 `role` 应使用该具体角色，如 `学P`、`燐羽`、`学园长`、`月村手毬`。角色名应尽量与领域知识库中的译名和简称一致。
- `楼主`：只用于楼主以论坛作者身份发出的说明、后记、更新告知等内容；不用于承载楼主楼里的所有正文。
- `回复者N`：用于 A 中对应非楼主回帖者的普通回帖内容。每个非楼主回帖者都是独立角色。
- C 的 `lines` 中默认不再写“角色名：”，因为角色已由 `role` 字段提供。只有当原文本身需要保留称呼或台词标签时，才保留在文本中。
- 如果同一语义角色的一段话需要拆成多条 cue，所有 cue 继续使用同一个 `role`，并在 `notes` 中标记 `split_from:Pxxxx`。

### 5.4 C 的分行与拆分规则

横屏字幕按以下规则切分：

- 默认每个 cue 只用 1 行。
- 每行约 15 到 20 个中文字符；不是硬性字数，但应避免明显过长。
- 只有当单行过长、且这句话的含义尚未表达完毕时，才拆成 2 行。
- 不需要按原文每一句都拆成一个 cue。短句可以合并，只要同一 cue 内仍是同一语义角色和同一阅读节拍。
- cue 也不应过短。只有 1 到 4 个字、单独看信息量不足的语气词、半句或反应词，原则上应并入同一角色的相邻 cue。
- 过短 cue 只有在原文刻意制造强停顿、情绪爆点、笑点、沉默或节奏反差时才单独保留。
- 优先在 `。`、`！`、`？`、`；`、`……` 后分段。
- 其次在 `，`、`、` 或自然语义停顿处分行。
- 不要产生只有 1 到 2 个字的孤立尾行，除非原文刻意如此。
- 同一角色一段话过长时，优先拆成多条 cue；只有单个意思单元内部过长时，才在同一 cue 中拆成 2 行。
- 一个 cue 内不混入多个角色。若翻译版 B 中连续出现不同角色内容，C 中必须拆成多个 cue。
- 楼主正文中叙述句和角色台词相邻时，也必须拆成 `旁白` 与具体角色两个 cue，不能统一归到 `楼主`。
- 字幕应适合快速阅读。可以调整中文语序，但不得删掉原文的关键信息。

## 6. 推荐仓库目录结构

为了支持多个翻译领域，建议将 Research-001 中的单一 `knowledge/` 调整为领域化目录：

```text
ForkTranslation/
├─ README.md
├─ docs/
│  ├─ research-001-gakumas-thread-translation-overview.md
│  ├─ research-002-animanch-rinha-unvideoized-threads.md
│  ├─ research-003-animanch-ss-research-workflow.md
│  └─ research-004-gakumas-thread-translation-formats-and-knowledge-layout.md
├─ shared/
│  ├─ schemas/
│  │  ├─ fork-thread-source-v1.md
│  │  └─ fork-pr-subtitle-v1.md
│  └─ prompt-templates/
│     └─ strong-agent-translation.md
└─ domains/
   └─ gakumasu/
      ├─ README.md
      ├─ stable/
      │  ├─ common-background.md
      │  ├─ style-guide.md
      │  ├─ glossary.csv
      │  ├─ characters.md
      │  ├─ character-profiles/
      │  │  ├─ rinami.md
      │  │  └─ rinha.md
      │  ├─ community-expressions.md
      │  ├─ translation-decisions.md
      │  └─ sources.md
      └─ threads/
         └─ board-0000000-short-title/
            ├─ source-A.md
            ├─ context-pack.md
            ├─ translation-B.md
            ├─ pr-subtitles-C.jsonl
            ├─ review-notes.md
            └─ final.md
```

`shared/` 只放跨领域通用的 schema、脚本约定和 prompt 模板。具体作品、人物、术语和社区表达放入 `domains/<domain>/`。

## 7. gakumasu 稳定知识文件

`domains/gakumasu/stable/` 至少应包含以下文件。

### 7.1 common-background.md

共通剧情背景知识。它不应写成百科全书，而应服务翻译判断。建议包含：

- 作品基本设定、学园结构、制作人与偶像的关系；
- 主要剧情阶段和重要事件；
- 常见剧情主题，如才能、竞争、亲爱度、组合活动、偶像路线等；
- 与论坛讨论高频相关的现实运营、活动和直播背景；
- 容易影响翻译判断的设定边界，例如哪些是官方设定，哪些是社区二创共识。

### 7.2 characters.md

所有角色的概要文件，但不能过度简略。每个角色建议包含：

- 中文名、日文名、常见简称、昵称和社区称呼；
- 基本身份、关系网和剧情位置；
- 说话风格、常见语尾、礼貌程度、情绪表达方式；
- 翻译时最容易错判的点；
- 是否已有独立详述文件。

该文件用于让强 agent 快速建立全局人物地图。

### 7.3 character-profiles/*.md

当某个翻译项目以一个或若干角色为主人公时，应为这些角色准备独立详述文件。每个文件建议包含：

- 角色核心性格与矛盾；
- 主线、个人剧情、活动剧情中的关键节点；
- 与本次串相关的人物关系；
- 典型台词、口癖和语气强度；
- 社区常见二创形象与官方形象的差异；
- 翻译口吻注意事项；
- 禁止强行脑补的边界。

例如燐羽中心串应加载 `characters.md` 与 `character-profiles/rinha.md`；如果同时大量涉及咲季或星南，则再按需加载对应详述文件。

### 7.4 style-guide.md

领域级翻译风格指南。建议明确：

- 简体中文、中文互联网语感、字幕可读性的取舍；
- 专名、称呼、敬语、粗口、语气词、括号和省略号规则；
- 梗、反讽、复读和错字的处理方式；
- 译注能否进入正文，何时只能进入备注；
- B 与 C 的差异：B 可保留较完整表达，C 必须为视频字幕切分。

### 7.5 glossary.csv

术语表。建议字段：

```csv
source,translation,category,status,notes
学園アイドルマスター,学园偶像大师,title,confirmed,可按项目风格简称学马仕
```

`status` 可使用 `confirmed`、`preferred`、`uncertain`、`deprecated`。

### 7.6 community-expressions.md

社区表达、黑话、跨圈梗和典型句式。每条建议包含：

- 原文表达；
- 直译含义；
- 社区语境；
- 推荐译法；
- 置信度；
- 示例。

### 7.7 translation-decisions.md

保存已经做过的翻译决策，尤其是有争议或可能反复出现的译法。每条建议包含：

- 决策日期；
- 问题表达；
- 采用译法；
- 不采用译法；
- 理由；
- 影响范围；
- 后续是否允许推翻。

## 8. 强 agent 翻译时的上下文包

每次翻译不应让强 agent 读取整个仓库。建议为每个串准备 `context-pack.md`，列出本次必须加载和可按需加载的文件。

推荐最小上下文：

- `domains/gakumasu/stable/common-background.md`
- `domains/gakumasu/stable/style-guide.md`
- `domains/gakumasu/stable/glossary.csv`
- `domains/gakumasu/stable/characters.md`
- 本次主角对应的 `character-profiles/*.md`
- 与本串相关的 `community-expressions.md` 条目
- 当前 `source-A.md`
- 当前串事件背景、疑点和人工备注

强 agent 的任务顺序：

1. 读取 A 和上下文包，确认楼层数、角色标签和缺失信息。
2. 先输出 B，保持与 A 的 `post` ID 对齐。
3. 对 B 做一次忠实度检查，确认没有漏译、加戏或擅自删减。
4. 从 B 生成 C，按 PR 字幕规则切分。
5. 对 C 做格式检查，确认每条 cue 默认 1 行、最多 2 行，角色字段合法，`source` 可追溯。
6. 输出疑点清单，供人工审校后更新 `translation-decisions.md` 或相关知识文件。

## 9. 审校清单

每个翻译项目至少检查：

- A 中每个 `post` 是否在 B 中出现；
- B 中每个译文是否能追溯回 A 的原文；
- C 中每条 cue 是否能追溯回 B 的楼层；
- 是否出现未在 A 中存在的新事实或新解释；
- 是否误把推测写成确定设定；
- 角色标签是否前后一致；
- 人物口吻是否符合 `characters.md` 和独立角色文件；
- 术语是否符合 `glossary.csv`；
- 长句是否已在 C 中拆成多条 cue；
- 每屏字幕是否默认 1 行、最多 2 行；
- 旁白是否对应原文中的叙述句、场景句或必要标题转场，没有替原文解释过度；
- 存疑内容是否进入备注或疑点清单，而不是混入正文。

## 10. 对 Research-001 的更新点

Research-001 中“翻译产品”“稳定知识”“目录结构”和“推荐工作流”的大方向仍然成立，但在当前需求下应更新为：

- 翻译输入不再假设由 agent 直接抓取网页，而是优先接收用户通过油猴脚本生成的标准文本 A。
- 知识库从单一 `knowledge/` 改为 `domains/<domain>/stable/`，以便容纳多个领域。
- `gakumasu` 作为独立领域目录，拥有自己的背景、术语、人物、社区表达和翻译决策。
- 输出不止一份最终译文，而是固定分为 B 和 C：B 用于审校，C 用于 PR 字幕脚本。
- C 的角色体系应从 A 的论坛作者标签进一步语义拆分：楼主正文中的叙述句为 `旁白`，角色台词为具体角色；楼主作者说明可用 `楼主`；每个非楼主回复者仍应成为可识别角色。
- 字幕切分是独立阶段，不能反过来破坏 B 的完整翻译与可追溯性。
- 稳定知识中必须有共通剧情背景文件、全角色概要文件，以及主角级独立详述文件。
