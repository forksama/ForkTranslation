# 调研档案 005：学马仕稳定知识库首版规格

- 档案编号：Research-005
- 建档日期：2026-07-13
- 状态：规格草案
- 主题：`domains/gakumasu/stable/` 各文件的首版规格与遗留待调研项

## 1. 本档案定位

Research-004 §7 定义了 `domains/gakumasu/stable/` 需要哪些稳定知识文件，但没有说明每份文件的**首版应该长什么样**。本档案在 Research-004 的基础上，把每份文件的首版规格逐一落定，包括收录范围、字段设计、粒度约束和信息来源。目的是让下一步能直接开始动手写这些文件，而不需要再反复决定"要不要"和"写多细"。

本档案的所有决策以"首版能启动第一批实际翻译"为目标，不追求一次性写全，也不预设未来一定按同一形态维护。

## 2. 跨文件通用前置

这些决定适用于 `stable/` 目录下的所有文件，不重复写进各文件规格中。

### 2.1 官方中文的存在性

《学园偶像大师》目前没有任何官方中文版本。所有中文译名均来自社区。因此本知识库不设"官方中文"来源级别，`status = confirmed` 只表示"社区共识稳定"，不表示"官方钦定"。

### 2.2 权威来源

在没有官方中文的前提下，本知识库以**中文汉化组/翻译 UP 主的既有输出**作为主要权威来源。具体人选和作品列表不在本档案锁定，需先做一轮交叉对比后再回填到 `sources.md`（延后创建）或术语表的 `notes` 中。

### 2.3 时效锚点

首版稳定知识锚定到 **2 周年（2026-05）**：主线剧情、现役担当、基础系统均以 2 周年周边已实装的版本为准。此后新增内容作为增量补充，不追溯改写首版。

### 2.4 主系列继承

对偶像大师主系列（765PRO、灰姑娘女孩、百万现场、SideM、闪耀色彩）已经形成的中文惯例（如"制作人""偶像""担当""亲爱度"等），**默认继承**。只在学马仕明确不同时才另行锁定。继承点不需要在本知识库重复登记，遇到冲突时再单独立条。

## 3. common-background.md 首版规格

### 3.1 收录范围

首版只写"骨架 + 高频事件"：

- **骨架**：学园结构、Sense/Logic 分支、亲爱度机制、组合与个人活动周期、P 与偶像的契约模型。
- **高频事件**：2 周年前主线剧情中翻译时反复被提及的关键节点，只写事件名称与最小必要说明，不复述剧情。

### 3.2 边界约束

- 只写**官方设定**。二创共识一律不进 common-background。二创层面的角色形象偏移写入对应 `character-profiles/*.md`；二创社区表达写入 `community-expressions.md`。
- 现实运营层（直播、卡池、活动、周年周边）**首版不写**。遇到具体串需要时，在该串的 `context-pack.md` 里临时补充。
- 不复述 wiki，只写"影响翻译判断"的内容。

## 4. characters.md 首版规格

### 4.1 收录范围

首版覆盖：

- 现役担当（可担当的偶像）；
- 关键 NPC（学园长、秘书等频繁出现的非担当角色）；
- 中等部/前史人物（SyngUp! 成员、学园旧生、父母及家人）。

具体角色清单不在本档案锁定，需在真正开始撰写时按上述范围点名。

### 4.2 每人固定字段

沿用 Research-004 §7.2 的建议字段（中文名、日文名、简称/昵称、社区称呼、身份关系、说话风格、翻译最易错判点、是否有独立 profile），但作以下调整：

- **人物关系**字段不在 characters.md 内展开，只留一个"主要关系对象名"清单，具体关系描述统一放到 `relationships.md`。
- **说话风格与口癖**、**官方剧情典型台词** 两项要额外从官方剧情（主线、卡剧情、亲爱度剧情）里抽样，不仅依赖 wiki 上的"口癖列表"拼凑。
- **社区形象与二创偏差**不进 characters.md，放到对应的 `character-profiles/*.md`。
- **翻译最易错判点** 首版留空，靠后续实际翻译过程中反推回填。

## 5. character-profiles/\*.md 首版规格

### 5.1 收录门槛

只为**当前或短期内计划翻译的项目主人公**准备独立 profile。首版只写一位：

- `character-profiles/rinha.md`（贺阳燐羽）

其他角色的 profile 一律**首版不建**，待真正出现以该角色为中心的翻译项目时再补。

### 5.2 密度上限

每份 profile 推荐 **800–1500 字**，**允许略超**（如为了完整覆盖官方设定而写到 2000 字左右，视为合理）。硬约束是"只写直接影响口吻与需要判断的信息"，不写剧情总结、不复述 wiki。字数不是目标而是提醒——写完自然到位就停，不因字数不足硬撑，也不因字数超一点强砍官方。

### 5.3 语料来源

- **主源**：官方剧情（主线、卡剧情、亲爱度剧情）中的台词与描写。
- **补源**：高质量二创作品中的台词与设定，允许引用，但必须在 profile 中显式标注"来自二创，非官方"。
- 不使用无来源的社区印象或 AI 推断。

### 5.4 二创偏差与脑补边界

- 每份 profile 的**内容重心在官方设定**（性格、说话风格、经典台词、主要关系）。二创部分**只举 2-3 个代表性精简例子**，不追求穷举。
- 每个二创倾向项只写"来源 + 一句翻译边界"即可，不做深度剧情复述。
- **不为写二创而压缩官方设定**。字数受限时先砍二创，再砍官方。
- "禁止脑补的边界"不单独开一节，作为二创偏差条目下的一行标记合并处理。

## 6. style-guide.md 首版规格

### 6.1 语感基线

译文中文语感向**日语宅圈 / 汉化组风格**靠。允许保留少量日式表达和宅圈术语不译，允许必要时加译注。不追求脱日化的自然中文，也不采用 B 站弹幕/微博风格。

### 6.2 称呼与敬语

- `プロデューサー / P → 学P`（作为角色称呼固定，不再逐次决定）。
- 对话中角色以名字称呼 P 时，可用"制作人"。
- 日文后缀 `さん / 様 / ちゃん / 先輩` 等**全部本土化**为中文习惯（如"小〇〇""〇〇前辈""〇〇小姐"），不保留原日文后缀。
- 官职（学园长、秘书等）直接中文化。

### 6.3 粗口与强语气

采用"偏向软化但不过度"的策略，按上下文情景判断强度：

- `バカ / ばか` 视语气强度译作"笨蛋""蠢货"等中等强度中文词，不译作"傻瓜"这类过软表达，也不拔高到"废物"以上。
- 更强等级（`クソ / 死ね` 等）视具体情景判断，允许保留粗口力度，不硬性统一。

### 6.4 语气词

日文句尾 `ね / よ / な / わ / かな / だろ / でしょ` 等**不逐一映射**到中文。目标是"中文读起来自然"，句尾语气靠句式、标点、词序传达，只在必要时加"呢""吧""啊""吗"。

### 6.5 符号规则

- 对话引号保留 `「」`（与 A/B/C 中"角色名「台词」"的写法统一，便于机器拆分）。
- 省略号统一使用中文全宽 `……`，不使用日文单字符 `…`，也不拓展为 `…………`。
- 人名中的中黑 `・` 不保留，改用 `·` 或直接空格。
- **书名号 / 作品名号按原文保留**：原文使用 `『』` 的（歌曲名、专辑名、作品名、剧情章节名等），译文也使用 `『』`；原文使用 `「」` 的（对话、强调、引用等），译文也使用 `「」`；原文使用 `《》` 或 `""` 的，译文分别沿用 `《》` 或 `""`。不做统一化替换，也不引入中文书名号 `《》` 覆盖原文的 `『』`。

### 6.6 梗与反讽

优先**保留日文原梗**，字幕或译文中必要时加译注。不主动替换成中文圈现有梗，也不为了流畅擅自脑补解释。无法判断时按"不加戏"原则直译，并在备注中标注为疑点。

### 6.7 B 与 C 差异实例

style-guide.md 内需要预留一段完整的**对照实例**：一段日文原文 + 对应的 B 版译文 + 对应的 C 版字幕 cue。这段实例作为"审校版 vs 字幕版"差异的锚点样例，帮助后续翻译时直接对齐。首版实例内容留白，在第一个真实翻译项目完成后回填。

## 7. glossary.csv 首版规格

### 7.1 category 取值

采用**粗粒度五大类**：

- `character`：角色专名（人名、别名、组合内成员）；
- `group`：组合、社团、势力名（如 SyngUp!、学园）；
- `content`：作品内命名内容（歌曲、卡、活动、剧情标题）；
- `system`：游戏系统术语（Sense/Logic、亲爱度、才能等）；
- `meme`：社区梗中作为"专名"稳定引用的（如社区自创组合别名、专有称呼）。

不细分到"歌曲 / 技能 / 活动"层级，等后期真的需要时再拆。

### 7.2 种子来源

首批术语从**中文汉化组和翻译 UP 主输出的交集**中抽取：优先选择多个来源一致的译名。不从日文官网/wiki 直接罗列后自行翻译，避免与中文圈已有共识冲突。

### 7.3 冲突登记

同一日文术语在不同中文来源有不同译法时，**主表只保留一行**（推荐译名），其他译法写入 `notes` 字段，格式如：

```csv
source,translation,category,status,notes
賀陽燐羽,贺阳燐羽,character,confirmed,"另见部分 UP 主译为“加阳燐羽”"
```

不为并存译名开多行，也不单独建 `deprecated-terms.md`。

### 7.4 粒度边界

- 招牌台词、口癖、角色固定文本 **不进** glossary。这些内容放到 `characters.md`（口癖）或 `community-expressions.md`（社区表达）。
- glossary 只收"可以当专名引用"的项。判断标准：这个词在别的句子里能作为专有名词直接出现。

## 8. community-expressions.md 首版规格

### 8.1 覆盖社区

首版只覆盖两个来源：

- **あにまん掲示板**（主要来源，与本项目翻译对象直接对应）；
- **B 站中文圈**（弹幕、评论区中已定型的中文圈表达）。

5ch、X（Twitter）不在首版覆盖范围内，遇到时按需补充。

### 8.2 分层策略

首版**只写学马仕独有 + 偶像大师系列已有共识**的社区表达。泛宅圈通用表达（如"概念""曇らせ""湿度"等）不进首版，遇到必要时再单独立条。

### 8.3 意图字段

**不新增独立字段**记录"发言意图"（反讽 / 复读 / 钓鱼 / 认真）。意图和使用语境自然融入 `context` 或 `notes` 字段的描述中。

### 8.4 置信度分级

置信度字段采用**三档**：

- `confirmed`：社区共识稳定，可以直接用于翻译；
- `probable`：高概率推断，需在译文附近做保守处理；
- `uncertain`：存疑，不应据此下结论，需人工判断或加译注。

### 8.5 种子条目

首版 20–30 条种子表达从两个渠道抽：

- Research-002 已经列出的**燐羽系列 SS 串**（作为主要语料来源）；
- **あにまん学马仕综合归纳串**（覆盖非燐羽中心的常见表达）。

## 9. translation-decisions.md 首版规格

### 9.1 预锁定决策

首版预先锁定以下决策：

- **`SyngUp!` 保留原文**，不中文化，不采用"震升！""声扬！"之类的中文化尝试。
- **担当名锚定原则**成立：所有担当中文名一旦在 glossary 中确定，即为该角色在全项目内的统一译名，不允许某个串内单独变体。**具体译名在 glossary.csv 中定，不在本文件重复登记**。
- `プロデューサー/P → 学P` 已在 style-guide 中锁定，不重复登记。
- `学園アイドルマスター → 学马仕` 是否作为项目内简称的锁定项，首版**不做决定**，等实际使用中出现分歧再补。

### 9.2 入库门槛

**不预先设定门槛**。是否入库由每条决策的性质单独判断：一次性局部选择、无争议惯例可以不入库；反复出现、有分歧、容易被下次翻译推翻的决策必须入库。

### 9.3 scope 字段

新增 `scope` 字段，固定取值：

- `global`：全领域适用，任何串都应遵守；
- `character`：只对指定角色生效；
- `domain-thread`：只对指定串生效（用于记录一次性局部决策，避免被误当全领域）。

## 10. relationships.md 首版规格（Research-004 未列的新增文件）

Research-004 §7 中不含此文件，本档案根据 §4 中"人物关系"字段的独立化需求新增。

### 10.1 定位

集中记录角色之间的关系描述，避免在 characters.md 与各 character-profile 中重复写。

### 10.2 字段建议

每条记录为"角色 A × 角色 B"的有序或无序对，包含：

- 双方角色 ID 或中文名；
- 关系类型标签（如亲属、同组合、师徒、单向倾慕、竞争等）；
- 一句中文自然语描述；
- 主要出处（官方剧情 / 社区二创）。

### 10.3 首版收录范围

只写以下两类关系：

- 首批 profile 涉及的角色（rinha）与其主要相关人物；
- characters.md 中"主要关系对象名"清单里出现的对；

其他关系遇到再补。

## 11. 首版**不**创建的文件

以下文件在首版一律不建，等实际需要时再创建：

- `sources.md`：外部信息源、快照、失效链接的登记；
- `songs.md` / `cards.md` / `events.md`：内容层拆文件；
- `honorifics-and-pronouns.md`：称呼与代词的独立文件；
- `deprecated-terms.md`：术语弃用记录（并入 glossary.csv 的 notes）。

#### 12.2.6 首版权威等级

`glossary.csv` 的 `status` 字段解释在本项目内做如下映射：

- **`probable`**：仅有 gakumas.cn 项目组或仅有独立源覆盖，但未见冲突。
- **`uncertain`**：多源之间存在明显冲突，或仅有单源覆盖且属于低播放量/低更新频率来源。
